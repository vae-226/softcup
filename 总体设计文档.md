# 总体设计文档

## 1. 项目概述

### 1.1 项目名称

灵境导游 - AI数字人景区导览系统

### 1.2 赛题来源

本项目对应第十五届中国软件杯大赛 A 组赛题“A5-景区导览服务AI数字人”，出题企业为锐捷网络（苏州）有限公司。赛题要求开发一个具备多模态交互能力的 AI 数字人导游软件，覆盖游客交互端和管理后台，并基于示范景区公开资料包进行开发与测试。

### 1.3 建设目标

构建一个面向灵山胜境示范景区的 AI 数字人导览系统，为游客提供实时智能问答、景点讲解、个性化路线推荐和情感化互动，并为景区管理者提供知识库维护、数字人配置、游客感受度分析和运营数据概览。

### 1.4 业务痛点

当前景区导览服务存在以下问题：

1. 高峰期专业导游资源不足，游客难以及时获取高质量讲解。
2. 传统录音导览内容固定，无法回答游客的个性化问题。
3. 普通设备缺少表情、语音和陪伴感，沉浸体验较弱。
4. 景区管理方难以量化导览服务质量，缺少游客反馈和运营优化依据。

## 2. 赛题要求对齐

| --- | --- |

| 智能问答与讲解 | 基于灵山胜境知识库进行景区历史、文化、景点特色和服务信息问答 |

## 3. 总体方案

系统分为游客交互端和管理后台端。

### 3.1 游客交互端

游客交互端面向景区游客，提供以下能力：

1. 文本提问和语音输入模拟。
2. 数字人语音、表情和口型状态联动展示。
3. 灵山胜境、灵山大佛、九龙灌浴、灵山梵宫、五印坛城等景点问答和讲解。
4. 根据游客兴趣推荐不同游览路线和讲解重点。
5. 根据当前景点切换讲解场景，并给出下一步追问建议。

### 3.2 管理后台端

管理后台端面向景区运营人员，提供以下能力：

1. 知识库管理：查看、启停、模拟上传景区讲解词、文史资料、常见问答等内容。
2. 数字人形象管理：配置数字人姓名、服装、声音、角色风格和讲解语气。
3. 游客感受度报告：分析游客反馈、满意度和情绪倾向，输出服务优化建议。
4. 数据大屏概览：展示服务人次、热门问答、满意度趋势、响应时间和准确率指标。

## 4. 系统架构设计

### 4.1 逻辑架构

系统分为四层：

1. 表现层  
   微信小程序游客导览页、景区终端页面、后续管理后台页面。

2. 能力层  
   文本理解、语音识别与合成接口、多模态大模型、知识检索、路线推荐、数字人状态驱动。

3. 数据层  
   景区知识库、游客画像、交互日志、反馈数据、数字人配置数据、运营指标数据。

4. 管理层  
   知识库运营、数字人配置、服务监控、满意度分析、可视化报表。

### 4.2 技术路线

本项目当前版本以可运行、易演示、便于视频录制为优先目标，采用原生微信小程序实现游客导览首页：

- 前端：原生微信小程序，使用 WXML、WXSS、JavaScript。
- 页面：当前实现 `pages/guide/guide` 游客导览首页。
- 核心模型：初赛小程序版本先使用本地知识库问答；后续通过 HTTPS 后端代理或微信云函数接入 DashScope / Qwen、Qwen-Omni、GLM-4.1V、GPT-4o 等模型。
- 语音识别：初赛版本为语音输入模拟；工程化版本可接入 Whisper、FunASR 或 Paraformer。
- 语音合成：初赛版本以前端状态展示为主；工程化版本可接入 Edge-TTS、ChatTTS 或商业 TTS。
- 数字人驱动：初赛版本用 2D 数字人状态和口型动画模拟；工程化版本可接入 Live2D、开源 2D 数字人 SDK 或 WebRTC 数字人服务。
- 知识库：初赛版本使用 `data/scenic.js` 中的本地结构化知识；工程化版本可升级为 MySQL/SQLite + 向量数据库 + Redis 缓存。

## 5. 示范景区资料包使用设计

赛题提供的“示范景区公开资料包”是本项目知识库和测试数据的基础来源。资料包包含：

1. `灵山胜境 景点结构化数据集.docx`  
   包含灵山胜境 16 个景点、拈花湾禅意小镇 6 个景点的结构化字段，如景点ID、景点名称、具体位置、建筑/景观参数、核心功能、文化内涵、详细介绍、游玩亮点、演艺/开放信息和备注。

2. `灵山胜境：历史、文化、景点特色与个性化游览指南.docx`  
   包含景区历史、佛教文化、核心景点详解、历史文化路线、自然风光路线、亲子家庭路线和实用游览贴士。

3. `景点景区旅游数据行为分析数据.xlsx`  
   包含游客画像、景点类型、游览日期、停留时长、消费、同行人数、满意度等游客行为字段，可用于后台运营分析和游客感受度报告。

当前原型已将上述资料中的核心内容整理到 `data/scenic.js`：

- `knowledgeBase` 保存灵山胜境概况、小灵山佛教缘起、灵山大佛、灵山梵宫、九龙灌浴与实用贴士等知识条目。
- `routes` 保存历史文化、自然风光、亲子家庭三类推荐路线。
- `analytics` 保存服务人次、响应时间、满意度、准确率、情绪趋势和游客反馈样例。
- `avatarProfiles` 保存数字人形象、服装、声音和讲解风格配置。

工程化版本中，资料入库流程为：

1. 解析 Word 和 Excel 文件，抽取景区文本、结构化景点字段和游客行为字段。
2. 按景点、主题、问答意图进行切分，保留来源文件和字段信息。
3. 对知识片段向量化并建立索引，同时保留关键词检索字段。
4. 游客提问时先召回相关知识，再由大模型生成导游口吻回答。
5. 后台上传新文档后重新切分、索引和启用，保证景区信息可更新。

## 6. 核心功能设计

### 6.1 智能问答与讲解

系统采用 RAG 检索增强问答流程：

1. 将景区资料按景点、主题和服务事项拆分为知识条目。
2. 对问题进行意图识别，判断是历史文化、自然风光、亲子服务、演出时间还是游客服务问题。
3. 从知识库中召回相关条目，优先使用当前景点关联知识。
4. 将游客问题、游客上下文和召回知识发送给大模型生成回答。
5. 返回文本回答，并同步生成数字人的表情、口型和讲解状态。

当未配置真实大模型接口时，系统使用本地知识库匹配逻辑生成回答，保证演示不中断。

### 6.2 个性化路线推荐

系统根据以下因素推荐路线：

- 游客兴趣：历史文化、自然风光、亲子家庭、游客服务。
- 游览时长：2小时、4小时、半日游或深度游。
- 同行类型：个人、家庭、研学团、老年游客等。
- 当前景点：根据游客所在点位推荐下一站。
- 当前时段：结合演出时间、夜游时段和拥挤程度给出建议。

当前原型提供三类路线：

1. 历史文化爱好者路线：突出小灵山佛教缘起、祥符禅寺、灵山大佛和五印坛城。
2. 自然风光游客路线：突出太湖风光、菩提大道、灵山大佛观景和禅意园林。
3. 亲子家庭路线：突出九龙灌浴、佛手广场、百子戏弥勒、梵宫和互动体验。

### 6.3 情感互动与数字人表现

数字人交互由回答内容、游客意图和场景状态共同驱动：

1. 历史文化问题使用沉思、讲解型状态。
2. 自然风光和拍照问题使用微笑、轻松状态。
3. 亲子和互动体验问题使用活泼、兴奋状态。
4. 服务咨询问题使用平稳、耐心状态。

初赛版本以 2D 数字人状态、口型动画和语音按钮模拟多模态效果。工程化版本可接入实时 TTS 音频、口型参数和表情动作驱动，提高自然度。

### 6.4 游客感受度分析

系统对游客反馈、满意度和交互记录进行统计，输出：

- 游客关注点分析：识别游客集中咨询的景点、路线、演出、服务设施等主题。
- 情感趋势报告：按日期或时段统计积极、中性、待改善反馈占比。
- 服务优化建议：根据低分反馈和高频问题提示知识库更新、导览动线优化或现场服务改进。
- 热门问答统计：帮助景区管理方发现游客最关心的问题。

### 6.5 管理后台模块

管理后台按赛题要求分为四个模块：

1. 知识库管理  
   展示知识条目标题、分类、关键词、状态和关联景点，支持模拟上传、启用和停用。

2. 数字人形象管理  
   支持配置数字人姓名、服装、声音、讲解风格和简介，使其更符合灵山胜境佛教文化和太湖风光气质。

3. 游客感受度报告  
   汇总游客反馈、满意度评分和情绪倾向，并生成运营建议。

4. 数据大屏概览  
   展示当日服务人次、本周服务人次、平均响应时间、满意度、问答准确率、情绪趋势和热门问题。

## 7. 非功能设计

### 7.1 性能目标

- 语音问答平均延迟控制在 5 秒以内。
- 景区事实问答准确率目标不低于 90%。
- 管理后台核心数据页面在本地环境下秒级打开。
- 演示过程不出现崩溃或长时间无响应。

### 7.2 自然度目标

- 数字人回答时有语音、表情、口型状态联动。
- 历史讲解、亲子互动、服务咨询等不同场景有不同表情状态。
- 回答风格保持真人导游语气，避免生硬问答。
- 演示视频中重点展示问答、路线推荐和数字人状态变化。

### 7.3 准确性与可控性

- 回答优先依据本地景区知识库，避免脱离资料包编造景区事实。
- 大模型系统提示限定回答范围，知识库没有的信息提示游客向景区服务中心确认。
- 后台知识条目可启停，便于管理方维护过期内容。
- 关键指标包括问答准确率、平均响应时间和满意度。

### 7.4 可扩展性

- 模型接口可替换，支持通过后端代理或云函数接入更强多模态模型。
- 知识库可从小程序本地 JavaScript 数据升级为数据库和向量索引。
- 数字人可从 2D 状态动画升级为 Live2D 或商业数字人 SDK。
- 可扩展多景区、多角色、多语言能力。

### 7.5 弱网与定位复杂场景

针对赛题鼓励考虑的复杂现实场景，系统可扩展以下策略：

1. 常见问题、基础讲解词和热门路线离线缓存。
2. GPS 信号弱时结合蓝牙 Beacon、二维码点位或景区终端点位切换。
3. 对重点区域预生成语音讲解和数字人动作，减少实时推理压力。
4. 网络异常时降级为本地知识库回答，保障基础导览可用。

## 8. 初赛版本实现说明

当前提交版本为初赛演示版本，重点展示产品闭环和赛题核心能力：

- 可在微信开发者工具中导入根目录运行。
- 当前版本先完成游客导览首页，覆盖数字人展示、本地知识库问答、路线推荐、百科磁贴、周边服务和讲解控制条。
- 后台页、真实 ASR/TTS、真实大模型接口和数字人 SDK 接入作为下一阶段扩展。
- 适合初赛答辩展示和不超过 7 分钟的演示视频录制。

当前版本的边界：

- 语音输入、TTS 和数字人口型为小程序前端演示模拟，不等同于完整实时音视频数字人 SDK。
- 游客行为分析使用资料包字段和样例数据进行展示，未连接生产数据库。
- 知识库上传为原型交互，工程化版本需补充文件解析、持久化和索引重建。
